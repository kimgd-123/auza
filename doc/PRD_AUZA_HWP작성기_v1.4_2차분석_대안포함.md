# AUZA PRD v1.4 2차 분석 메모 (대안 포함)

- 작성일: 2026-03-18
- 분석 대상:
  - `doc/PRD_AUZA_HWP작성기.md`
  - `doc/PRD_AUZA_HWP_writer_analysis_for_claude.md`
- 참고 전제:
  - 현재 워크스페이스에는 구현 코드가 거의 없고, PRD/분석 문서 중심 상태다.
  - 따라서 본 문서는 "현재 PRD 문서 자체의 정합성/구현 리스크/대안"을 검토한 결과다.

## 1. 총평

PRD는 이전 버전보다 분명히 좋아졌다. 특히 아래 항목은 방향이 맞다.

- PDF 표를 "직접 붙여넣기"가 아니라 `Gemini Vision 기반 재구성`으로 재정의한 점
- `자유 채팅`과 `구조화 작업`을 분리한 점
- 표 생성에 대해 `JSON 출력 계약`을 넣은 점
- `v1 비목표`, `중복 삽입 정책`, `HWP 연결 정책`을 명시한 점

다만 현재 버전은 기능 아이디어는 좋아졌지만, 실제 구현 단계에서 가장 많이 흔들릴 부분이 아직 남아 있다. 핵심은 아래 5개다.

1. Vision 결과와 채팅 결과를 어떤 공통 데이터 모델로 다룰지 불명확함
2. TipTap/HTML/LaTeX 중 무엇을 진짜 원본 데이터로 볼지 정해지지 않음
3. PDF 캡처 품질과 좌표 처리 방식이 아직 구현 수준으로 내려오지 않음
4. 수식 파이프라인이 v1 범치고는 무겁고 실패 지점이 많음
5. 세션 저장/복구가 빠져 있어 실제 업무 도구로 쓰기엔 불안정함

## 2. 핵심 이슈와 대안

### 이슈 1. 문서 메타 정보와 섹션 번호가 일부 어긋남

현재 PRD에는 아래 불일치가 있다.

- 문서 상단 버전은 `v1.2`인데 변경 이력은 `v1.4`까지 반영되어 있음
- `## 9.`가 두 번 등장함
- `### 4.4`가 에디터 블록과 Gemini 채팅에서 중복 사용됨

이건 기능 이슈는 아니지만, Claude나 이후 구현자가 문서를 참조할 때 혼선을 만든다.

대안:

- 대안 A: PRD 버전을 `v1.4`로 올리고 섹션 번호를 전부 정리한다
- 대안 B: 변경 이력만 남기고 상단 버전은 마지막 승인 시점까지 고정한다

권장안:

- `대안 A`
- 지금 문서는 초안이 아니라 구현 직전 설계 문서에 가까워졌기 때문에, 참조 포인트가 흔들리면 바로 작업 지시 품질이 떨어진다

### 이슈 2. Vision 경로와 채팅 경로가 서로 다른 형식으로 에디터에 들어감

현재 PRD는 두 경로를 사용한다.

- 캡처 경로: `Gemini Vision -> HTML + LaTeX 혼합 -> 에디터 삽입`
- 구조화 채팅 경로: `Gemini -> JSON -> 검증 -> HTML 표 변환 -> 에디터 삽입`

이 구조는 당장 문서상 설명은 가능하지만, 실제 구현에서는 두 문제가 생긴다.

- 같은 "표"인데 어떤 것은 HTML 직접 삽입, 어떤 것은 JSON 검증 후 삽입이 됨
- 나중에 HWP 변환 시 파서가 어느 포맷을 기준으로 신뢰해야 하는지 흐려짐

대안:

- 대안 A: Vision과 채팅 모두 공통 `AUZA 구조화 모델(JSON)`로 변환한 뒤, 에디터 렌더링도 동일 경로로 처리
- 대안 B: Vision은 자유형 HTML을 허용하되, 에디터 삽입 전에 공통 정규화기에서 `중간 AST`로 바꾼 뒤 다시 렌더링
- 대안 C: 지금처럼 이중 경로를 유지하고 각 경로마다 별도 파서를 둠

권장안:

- 단기: `대안 B`
- 중장기: `대안 A`

이유:

- 지금 단계에서 Vision까지 완전 JSON 강제는 프롬프트 난이도가 높다
- 대신 "Vision도 바로 에디터에 넣지 않고 공통 정규화기 한 번 통과"만 해도 안정성이 크게 올라간다

### 이슈 3. 원본 데이터(Source of Truth)가 정의되지 않음

문서에는 아래 표현이 함께 나온다.

- `TipTap HTML`
- `HTML + LaTeX`
- `KaTeX 렌더링`
- `html_parser.py -> DocumentStructure`

하지만 실제 구현에서는 "무엇을 저장하고, 무엇을 렌더링하고, 무엇을 HWP로 변환할지" 하나로 정해야 한다.

이걸 안 정하면 아래 문제가 생긴다.

- TipTap이 수식/표를 내부적으로 표현하는 방식과 HTML 직렬화 결과가 달라질 수 있음
- 에디터에서 보이는 내용과 HWP로 나가는 내용이 다를 수 있음
- 채팅 수정 결과를 어느 레이어에 적용할지 불명확해짐

대안:

- 대안 A: `ProseMirror JSON`을 에디터의 원본 데이터로 사용하고, HWP 내보낼 때 `DocumentStructure`로 변환
- 대안 B: 앱 내부 공통 원본을 `DocumentStructure JSON`으로 두고, 에디터는 이를 렌더링하는 뷰로만 사용
- 대안 C: HTML 문자열을 원본 데이터로 간주

권장안:

- `대안 A`

이유:

- TipTap 기반 편집기라면 에디터 상태의 원본은 ProseMirror JSON이 가장 자연스럽다
- 대신 내보내기 직전에 `DocumentStructure`로 정규화하는 2단 구조가 현실적이다
- HTML은 뷰/복사/내보내기 포맷으로만 쓰는 편이 안전하다

### 이슈 4. PDF 영역 캡처 구현이 생각보다 까다로운데, PRD에는 품질 기준이 없음

PRD는 `Canvas API`로 영역을 캡처한다고 적고 있지만, 구현 단계에서는 아래 문제가 생긴다.

- 줌 비율에 따라 캡처 해상도가 달라짐
- 스크롤된 위치와 실제 PDF 좌표를 맞춰야 함
- 멀티 페이지 경계에서 드래그하면 처리 규칙이 필요함
- 스캔 PDF는 저해상도 캡처 시 Vision 인식률이 급격히 떨어질 수 있음

대안:

- 대안 A: 화면에 보이는 DOM/Canvas를 그대로 잘라서 Vision에 전달
- 대안 B: 사용자의 드래그 좌표를 pdf.js 페이지 좌표로 변환한 뒤, 더 높은 스케일로 오프스크린 재렌더링 후 캡처
- 대안 C: 페이지 전체 이미지를 고해상도로 만든 뒤, 서버/로컬에서 crop 처리

권장안:

- `대안 B`

추가로 PRD에 넣으면 좋은 조건:

- 캡처는 기본 2x 또는 3x 스케일로 생성
- 한 번에 한 페이지 안에서만 드래그 허용
- 인식 실패 시 `더 크게 확대해서 다시 캡처` 안내
- 스캔 PDF는 "지원"이 아니라 "캡처 품질에 따라 인식 정확도 달라짐"이라고 표현

### 이슈 5. 표 수정의 출력 계약은 생겼지만 "수정 방식"은 아직 불명확함

현재 예시는 좋다. 다만 `"3번째 열 삭제"` 같은 요청에서 모델이 무엇을 반환해야 하는지는 아직 분명하지 않다.

가능한 방식은 두 가지다.

- 전체 표를 다시 반환
- 변경 연산만 반환

대안:

- 대안 A: v1에서는 항상 `전체 표 스냅샷 JSON`을 반환
- 대안 B: `delete_column`, `merge_cells` 같은 연산 패치를 정의

권장안:

- `대안 A`

이유:

- v1에서 패치 연산까지 열면 검증기와 충돌 해결 로직이 바로 복잡해진다
- 전체 스냅샷 교체 방식은 단순하고 디버깅도 쉽다

PRD에 추가하면 좋은 문장:

> 구조화 작업의 결과는 항상 "적용 후 전체 표 상태"를 나타내는 JSON 스냅샷으로 반환한다. 부분 연산 패치는 v1 범위에 포함하지 않는다.

### 이슈 6. 수식 파이프라인은 매력적이지만, 현재 v1 범위에서는 가장 무거운 축임

현재 PRD의 수식 축은 아래 단계로 이어진다.

`Vision -> LaTeX -> normalizer -> KaTeX -> AST -> HWP EquEdit -> width fix`

이건 기술적으로 가능할 수는 있어도 실패 지점이 많다.

- Vision이 애매한 LaTeX를 반환할 수 있음
- normalizer에서 보정 안 되는 케이스가 남을 수 있음
- KaTeX가 해석 못 하는 문법이 나올 수 있음
- HWP EquEdit가 일부 표현을 다르게 렌더링할 수 있음
- width fix는 후처리 UX 이슈가 생길 수 있음

대안:

- 대안 A: v1에서 수식은 우선 이미지 삽입 fallback, 실제 LaTeX -> HWP 변환은 v1.1
- 대안 B: v1에서 지원 LaTeX subset만 명시하고, 미지원 문법은 이미지 fallback
- 대안 C: 현재 PRD처럼 전 범위를 시도

권장안:

- 일정이 촉박하면 `대안 A`
- 수식을 꼭 포함해야 하면 `대안 B`

현실적으로는 `대안 B`가 가장 균형이 좋다.

PRD에 추가하면 좋은 문장:

> v1의 수식 변환은 분수, 루트, 첨자, 합/적분 등 주요 학술 수식을 우선 지원하며, 미지원 LaTeX 환경은 이미지 삽입 또는 수동 보정 대상으로 처리한다.

### 이슈 7. HWP 실행 정책은 정리됐지만, 설정 항목은 아직 그 정책을 완전히 따라가지 않음

현재 PRD는 v1에서 `GetActiveObject` 중심 정책으로 정리됐다. 그런데 설정에는 아직 `HWP 경로`가 남아 있다.

이 상태에서는 아래 질문이 생긴다.

- 자동 실행을 안 한다면 HWP 경로는 왜 필요한가?
- `Dispatch`는 구현 참고용인지, 실제 v1 동작인지?

대안:

- 대안 A: v1 설정에서 `HWP 경로`를 제거
- 대안 B: `고급 설정(향후 기능)`으로 숨겨두고 문서에만 남김
- 대안 C: 다시 자동 실행 기능을 v1에 넣음

권장안:

- `대안 A`

PRD를 더 깔끔하게 만들려면 아래처럼 가는 편이 낫다.

- v1: 실행 중 HWP 연결만 지원
- v1.1 이후: HWP 자동 실행/새 문서 생성 지원

### 이슈 8. 실제 업무용 도구로 보려면 세션 저장/복구가 필요함

현재 문서에는 아래가 없다.

- 에디터 블록 임시 저장
- 채팅 히스토리 복구
- 캡처 결과 재열기
- 앱 비정상 종료 시 복구

이건 구현 난도가 높은 기능은 아니지만, 사용자 만족도에는 매우 크다. 특히 내부 직원 도구라면 더 그렇다.

대안:

- 대안 A: `.auza` 프로젝트 파일 저장/열기 + 자동 저장
- 대안 B: 마지막 작업 세션만 로컬 자동 저장
- 대안 C: v1 비목표로 명시하고 완전히 제외

권장안:

- 최소 `대안 B`
- 가능하면 `대안 A`

아무것도 안 넣을 거면 PRD에 최소한 아래를 명시해야 한다.

> v1은 프로젝트 저장/복구 기능을 제공하지 않으며, 앱 종료 전 사용자가 최종 결과를 HWP로 내보내는 것을 기본 사용 방식으로 한다.

### 이슈 9. v1 범위가 여전히 큼

현재 v1에는 아래가 모두 들어 있다.

- Electron/React/Tailwind 3패널 앱
- PDF 뷰어
- 드래그 캡처
- Gemini Vision
- TipTap 표/수식 편집
- 구조화 채팅
- LaTeX 정규화
- KaTeX 렌더링
- HWP COM 자동화
- 수식 후처리
- exe 패키징

이 범위는 아이디어로는 좋지만, 첫 배포 기준으로는 꽤 무겁다.

대안:

- 대안 A: `내부 알파`
  - 텍스트 복사
  - 캡처 기반 표 인식
  - 기본 표/HWP 삽입
  - 수식은 제외 또는 이미지 fallback
- 대안 B: `균형형 v1`
  - 텍스트 + 표 + 제한된 수식
  - 표는 JSON 검증
  - 수식은 whitelist + fallback
  - 저장/복구 최소 기능 포함
- 대안 C: 현재 PRD 범위를 그대로 v1에 포함

권장안:

- `대안 B`

이유:

- 지금 문서 방향을 크게 흔들지 않으면서도 일정 폭주를 막을 수 있다
- 특히 `표 구조화`와 `수식 자동화`를 같은 품질 수준으로 동시에 끝내려 하면 일정 예측이 매우 어려워진다

## 3. 권장 v1 재구성안

현재 PRD 방향을 최대한 살리면서 구현 가능성을 높이려면 아래 구성이 좋다.

### 권장안: Balanced v1

- 텍스트:
  - 선택 도구 + Ctrl+C/Ctrl+V 유지
- 표:
  - 영역 캡처 사용
  - Vision 결과는 바로 HTML 삽입하지 않고 공통 정규화기로 통과
  - 구조화 작업은 전체 표 JSON 스냅샷으로만 처리
- 수식:
  - LaTeX subset 지원
  - 미지원 문법은 이미지 fallback 또는 미지원 안내
- 에디터 데이터:
  - 원본은 ProseMirror JSON
  - 내보내기 직전 DocumentStructure로 변환
- HWP:
  - 실행 중 인스턴스 연결만 지원
  - 자동 실행/새 문서 생성은 제외
- 저장:
  - 최소 마지막 세션 자동 저장

## 4. Claude에게 바로 요청하면 좋은 수정 항목

1. 문서 상단 버전을 `v1.4`로 수정하고 섹션 번호를 정리
2. `공통 콘텐츠 모델` 섹션 추가
3. Vision 결과도 에디터 직삽입이 아니라 `공통 정규화기`를 거친다고 명시
4. 구조화 표 수정은 `전체 표 스냅샷 반환`으로 고정
5. PDF 캡처 품질 규칙(최소 스케일, 페이지 단위 제한, 재캡처 안내) 추가
6. 수식 지원 범위를 whitelist + fallback 정책으로 명시
7. v1 설정에서 `HWP 경로` 제거 또는 후순위 기능으로 이동
8. 세션 저장/복구 정책 추가

## 5. 최종 결론

현재 PRD는 방향성 자체는 상당히 좋아졌고, 특히 "AI 재구성" 관점은 이전보다 훨씬 설득력이 있다. 다만 지금 상태로 바로 구현에 들어가면 가장 먼저 흔들릴 부분은 UI가 아니라 `데이터 모델`, `캡처 품질`, `수식 범위`, `세션 저장`이다.

가장 실무적인 다음 단계는 아래 순서다.

1. 문서 메타/번호 정리
2. 공통 콘텐츠 모델 정의
3. 표/수식 각각의 fallback 정책 명시
4. Balanced v1 범위로 고정

이 네 가지가 정리되면, Claude가 작성할 다음 개정 PRD는 구현 착수용 문서로 훨씬 안정적이 된다.
