# AUZA PRD 남은 이슈 메모 (Claude 전달용)

- 작성일: 2026-03-18
- 대상 문서: `doc/PRD_AUZA_HWP작성기.md`
- 기준 버전: 4.3.4 캡처 품질 규칙 반영 이후 버전
- 목적: 이전 피드백이 대부분 반영된 상태에서, 아직 남아 있는 보완 포인트만 정리

## 1. 현재 상태 요약

4.3.4 `캡처 품질 규칙`은 전반적으로 잘 반영되었다.

- 고정 품질 확보를 위한 `2x~3x 오프스크린 재렌더링`
- pdf.js 기반 `화면 좌표 ↔ PDF 좌표 변환`
- `한 페이지 안에서만` 드래그 허용
- 인식 실패 시 `다시 캡처` 안내
- 스캔 PDF 정확도 한계 고지

즉, 이전에 지적했던 큰 방향은 대부분 정리되었다.

## 2. 남은 이슈

### 2.1 인식 실패 판정 기준이 아직 모호함

현재 PRD에는 아래처럼 적혀 있다.

> Vision 인식 결과가 비어있거나 신뢰도가 낮으면 "더 크게 확대해서 다시 캡처해주세요" 안내

문제는 `신뢰도가 낮다`는 표현이 구현 기준으로는 모호하다는 점이다.

- Gemini Vision이 항상 명시적인 confidence 값을 주는 것은 아님
- 구현자가 임의 기준을 만들 가능성이 있음
- QA 시 어떤 경우를 실패로 볼지 합의가 어려움

Claude가 PRD에서 더 구체화해야 할 내용:

- 응답이 비어 있는 경우
- HTML/표/수식 정규화 결과가 비어 있는 경우
- 표 인식 결과에서 행/열 구조가 성립하지 않는 경우
- LaTeX 파싱 또는 정규화 실패
- 사용자가 수동으로 `다시 캡처`를 누를 수 있는지 여부

권장 문구:

> 인식 실패는 Vision 응답이 비어 있거나, 표/수식 정규화 결과가 유효하지 않을 때로 정의한다. 예를 들어 표의 행/열 구조가 성립하지 않거나, LaTeX 정규화/파싱에 실패한 경우 실패로 간주한다.

### 2.2 2x~3x 스케일 범위는 있지만 선택 규칙이 없음

현재 PRD는 캡처를 `2x~3x 스케일`로 재렌더링한다고 적고 있다.

이 표현은 방향은 맞지만, 구현 단계에서는 아래 질문이 생긴다.

- 기본값은 2x인지 2.5x인지 3x인지
- 작은 영역은 더 높은 배율을 쓸지
- 너무 큰 영역은 메모리/속도 때문에 낮춰야 하는지
- Vision 전송 전에 최대 이미지 크기 제한이 있는지

이 기준이 없으면 환경에 따라 속도와 인식 품질이 들쭉날쭉해질 수 있다.

Claude가 PRD에서 더 구체화해야 할 내용:

- 기본 스케일
- 작은 영역/큰 영역에 대한 조정 규칙
- 최대 해상도 또는 긴 변 기준
- 필요 시 압축/리사이즈 여부

권장 문구:

> 캡처 기본 스케일은 2.5x로 하며, 작은 영역은 3x까지 허용한다. 단, 렌더링 결과의 긴 변이 임계값을 초과하면 2x로 낮춰 처리한다.

## 3. Claude가 바로 반영하면 좋은 수정안

### 3.1 4.3.4 인식 실패 안내 문구 보강

아래처럼 바꾸는 것이 좋다.

```md
| **인식 실패 안내** | Vision 응답이 비어 있거나, 표/수식 정규화 결과가 유효하지 않으면 실패로 간주. 예: 표 행/열 구조 불성립, LaTeX 파싱 실패. 이 경우 "더 크게 확대해서 다시 캡처해주세요" 안내 및 재시도 버튼 제공 |
```

### 3.2 4.3.4 캡처 스케일 규칙 보강

아래처럼 바꾸는 것이 좋다.

```md
| **캡처 스케일** | 사용자 드래그 좌표를 pdf.js 페이지 좌표로 변환 후, 기본 2.5x 스케일로 오프스크린 재렌더링하여 캡처. 작은 영역은 3x까지 허용하고, 결과 이미지의 긴 변이 임계값을 초과하면 2x로 낮춰 처리 |
```

## 4. 최종 판단

현재 PRD는 4.3.4 추가로 인해 캡처 품질 규칙이 상당히 안정되었다. 지금 남은 건 방향 수정이 아니라 `판정 기준의 명시`와 `스케일 결정 규칙의 고정`이다.

즉, Claude가 이번에 더 손봐야 할 부분은 크지 않다.

- 인식 실패를 어떻게 판단할지
- 2x~3x 중 실제 어떤 규칙으로 선택할지

이 두 가지만 명확해지면, 4.3.4는 구현 지시용 문서로 충분히 쓸 수 있다.
