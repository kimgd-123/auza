# v2.1.9 Python Embed 번들링 — 사이드이펙트 및 해결 기록

> 작성일: 2026-04-07
> 버전: v2.1.6 → v2.1.9
> 목적: Python 미설치 PC에서 AUZA 앱이 정상 동작하도록 Python Embeddable을 번들링

---

## 1. 배경

테스터 PC에서 `No module named 'huggingface_hub'`, `Python 프로세스를 시작할 수 없습니다` 에러 발생.
원인: 테스터 PC에 Python이 설치되어 있지 않음.

## 2. 적용한 해결책

| 구분 | 내용 |
|------|------|
| 설치 버전 | Python 3.12.9 Embeddable을 `extraResources`로 빌드에 포함 |
| 무설치 버전 | python-embed 미포함, 시스템 Python 또는 자동 다운로드 |
| OD 패키지 | 앱 시작 시 torch CPU + doclayout-yolo 자동 설치 (첫 실행 5~10분) |

---

## 3. 발생한 사이드이펙트 및 해결

### 3.1 빌드 시 vite build 누락

**증상**: 테스터 PC에서 GUI 버전이 2.1.6으로 표시, Python 변경사항 미반영

**원인**: `npx electron-builder`만 실행하고 `npm run build` (tsc + vite build)를 먼저 실행하지 않음. `dist/`, `dist-electron/`에 이전 빌드 결과물이 그대로 패키징됨.

**해결**: 반드시 `npm run build` → `npx electron-builder` 순서로 실행.

**예방**: `package.json`에 `electron:build:installer`, `electron:build:portable` 스크립트 추가하여 빌드 절차를 통합.

---

### 3.2 CUDA torch DLL 로딩 실패 (WinError 1114)

**증상**: OD 실행 시 `[WinError 1114] DLL 초기화 루틴을 실행할 수 없습니다. Error loading "...torch\lib\c10.dll"`

**원인 1 — CUDA torch 설치**: pip의 기본 torch는 CUDA 버전. NVIDIA GPU 드라이버가 없는 PC에서 c10.dll 로딩 실패.

**원인 2 — VC++ Runtime 버전 부족**: CPU torch도 최신 VC++ Runtime 필요. 테스트 PC의 vcruntime140.dll v14.32가 torch 2.11.0에 부족했음.

**해결**:
- `_ensure_od_packages()`에서 CPU torch를 `--index-url https://download.pytorch.org/whl/cpu`로 강제 설치
- `torch.version.cuda`를 체크하여 CUDA 버전이면 제거 후 CPU로 교체
- `except Exception`으로 ImportError뿐 아니라 OSError(DLL 실패)도 catch
- 테스터에게 VC++ Redistributable 최신 버전 설치 안내: `https://aka.ms/vs/17/release/vc_redist.x64.exe`

**예방**: 환경 진단 스크립트(`Check-AuzaEnv.ps1`)로 사전 확인. VC++ Runtime 버전이 v14.38 미만이면 업데이트 안내.

---

### 3.3 DLL 검색 경로 누락

**증상**: VC++ Runtime이 설치된 PC에서도 c10.dll 로딩 실패

**원인**: embed Python의 `vcruntime140.dll`이 `python-embed/` 루트에 있지만, torch의 c10.dll은 `site-packages/torch/lib/`에서 로드되어 DLL 검색 경로에 embed 루트가 포함되지 않음.

**해결**:
- `python-bridge.ts`: spawn 시 embed 디렉토리를 `PATH` 환경변수에 추가
- `detector.py`: `os.add_dll_directory()`와 `PATH` 환경변수 양쪽에 Python 디렉토리 추가

```python
# detector.py
def _add_dll_search_paths():
    python_dir = os.path.dirname(sys.executable)
    if hasattr(os, 'add_dll_directory'):
        os.add_dll_directory(python_dir)
    os.environ['PATH'] = python_dir + ';' + os.environ.get('PATH', '')
```

### 3.4 Portable 버전 실행 시간 급증

**증상**: 무설치(portable) exe 실행 시 5분 이상 소요

**원인**: portable exe는 매 실행마다 전체 리소스를 임시 폴더에 압축 해제. python-embed(78MB+)가 포함되면 해제 시간이 크게 증가.

**해결**: portable용 별도 빌드 설정(`electron-builder-portable.yml`) 생성, python-embed를 `extraResources`에서 제외. portable에서는 시스템 Python 또는 자동 다운로드 방식 사용.

---

### 3.5 python-embed 오염 (테스트 시 패키지 잔재)

**증상**: 빌드 크기가 1.2GB 이상으로 비정상 증가

**원인**: 개발 중 embed Python에서 `pip install torch doclayout-yolo` 등을 테스트한 후, 해당 패키지가 `python-embed/Lib/site-packages/`에 남아 빌드에 포함됨.

**해결**: 오염된 `python-embed/` 삭제 후 깨끗한 환경에서 재구성 (bs4 + pywin32 + Pillow + pip + google-generativeai만 포함).

**예방**:
- `python-embed/`는 `.gitignore`에 추가하여 실수 방지
- 테스트 시 embed Python 직접 사용 금지. 별도 경로(예: `C:\tmp\py-embed`)에서 테스트
- 빌드 전 `du -sh python-embed/` 로 크기 확인 (정상: ~240MB 이하)

---

### 3.6 이전 설치 잔재로 인한 반복 에러

**증상**: 새 빌드 설치 후에도 동일 에러 반복

**원인**: NSIS `oneClick: true` 설치는 기존 폴더를 완전히 삭제하지 않고 덮어씀. 이전 빌드에서 설치된 CUDA torch 등이 `site-packages/`에 잔존.

**해결**: 테스터에게 기존 설치 폴더를 수동 삭제하도록 안내:
```
C:\Users\{user}\AppData\Local\Programs\auza\
C:\Users\{user}\AppData\Local\Programs\auza-updater\
C:\Users\{user}\AppData\Roaming\AUZA-v2\
```

**예방**: `_ensure_od_packages()`에서 `torch.version.cuda` 체크 + `--force-reinstall`로 CUDA → CPU 자동 교체. 환경 진단 스크립트에서 AUZA 잔재 경로 확인.

---

### 3.7 google-generativeai 누락

**증상**: OD 인식 성공 후 `No module named 'google'` 에러

**원인**: embed Python에 `google-generativeai` 패키지를 번들하지 않음. OD 분석 후 Gemini API 호출 시 실패.

**해결**: 
- `python-embed/`에 `google-generativeai` 사전 설치
- `main.py`의 `_ensure_packages()`에 `google.generativeai` 추가

---

### 3.8 OD 패키지 설치 타임아웃

**증상**: OD 첫 실행 시 "요청 시간이 초과되었습니다" 에러

**원인**: torch CPU 다운로드/설치에 5~10분 소요되나, IPC 타임아웃이 5분(300초)으로 설정됨.

**해결**: OD 관련 IPC 타임아웃을 15분(900초)으로 확대.

---

### 3.9 앱 시작 시 이벤트 타이밍 문제

**증상**: OD 패키지 설치 진행상황이 화면에 표시되지 않음

**원인**: `main.ts`에서 `webContents.send()`가 React 컴포넌트 마운트 전에 호출되어 이벤트를 놓침.

**해결**: `webContents.once('did-finish-load')` + 1초 딜레이로 renderer 로드 완료 후 이벤트 발송. PdfViewer에 별도 배너 UI 추가.

---

### 3.10 개발 모드에서 embed Python 사용

**증상**: 개발 PC에서도 "PyTorch CPU 설치 중..." 표시

**원인**: `findPythonExecutable()`이 개발 모드에서도 프로젝트 루트의 `python-embed/python.exe`를 1순위로 선택.

**해결**: `app.isPackaged`가 true일 때만 번들 embed Python 사용. 개발 모드에서는 시스템 Python 우선.

```typescript
if (app.isPackaged) {
    const bundledPath = path.join(process.resourcesPath, 'python-embed', 'python.exe')
    if (fs.existsSync(bundledPath)) return bundledPath
}
```

---

### 3.11 python-bridge의 동기 호출로 앱 멈춤

**증상**: 무설치 버전에서 Python 자동 다운로드 중 앱이 "응답 없음"

**원인**: `python-installer.ts`에서 `execFileSync` (동기 호출) 사용. Electron 메인 프로세스의 이벤트 루프가 차단됨.

**해결**: `execFileSync` → `execFileAsync` (promisify된 비동기) 전환.

---

## 4. 환경 진단 스크립트

테스터 PC 문제 진단을 위해 `scripts/Check-AuzaEnv.ps1` + `scripts/AUZA_환경진단.bat` 생성.

**체크 항목**: OS, CPU, RAM, Disk, Python, VC++ Runtime, HWP, HWP COM, Internet, AUZA 잔재

**사용법**: bat 파일 더블클릭 → 결과 자동 클립보드 복사 → Ctrl+V로 공유

**주의**: ps1 파일은 반드시 UTF-8 BOM 인코딩으로 저장해야 한글 깨짐 방지.

## 5. 빌드 절차 요약

```bash
# 1. 프론트엔드 + Electron 빌드
npm run build

# 2-a. 설치 버전 (python-embed 포함)
npx electron-builder --win nsis --config electron-builder.yml

# 2-b. 무설치 버전 (python-embed 미포함)
npx electron-builder --win portable --config electron-builder-portable.yml

# 3. 분류
mkdir -p release/설치버전 release/무설치버전
mv "release/AUZA-v2 Setup *.exe" release/설치버전/
mv "release/AUZA-v2-*-portable.exe" release/무설치버전/
```
